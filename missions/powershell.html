Contexte

Objectif de la mission

Dans le cadre du support N1 chez VNF, certaines tâches d'administration répétitives — comme la création de comptes utilisateurs ou la réinitialisation de mots de passe — sont réalisées plusieurs fois par semaine. L'objectif est d'automatiser ces opérations via des scripts PowerShell interactifs pour gagner en rapidité, réduire les erreurs de saisie manuelle et standardiser les procédures.

Les scripts sont développés et testés sur Windows PowerShell ISE puis déployés sur les postes des techniciens du Helpdesk VNF.

Cas d'usage identifiés

Les scripts développés répondent à des besoins concrets du quotidien du Helpdesk VNF, liés à la gestion des comptes dans l'Active Directory.

👤
Création d'un compte utilisateur AD
Automatise la création d'un nouveau compte dans l'Active Directory : saisie du nom, prénom, service, définition du mot de passe et affectation aux groupes de sécurité.
New-ADUser
🔑
Réinitialisation de mot de passe
Recherche un utilisateur par son login, réinitialise son mot de passe selon la politique de sécurité VNF et force le changement à la prochaine connexion.
Set-ADAccountPassword
🔒
Déverrouillage de compte
Déverrouille rapidement un compte bloqué suite à plusieurs tentatives de connexion incorrectes.
Unlock-ADAccount

Procédure de création et déploiement

1

Identification et analyse du besoin

Avant d'écrire une seule ligne de code, il est essentiel d'analyser précisément le besoin pour définir ce que le script doit faire, quelles données il doit demander à l'utilisateur et quelles actions il doit effectuer dans l'AD.

1
Recenser les tâches répétitives
Identifier les actions réalisées manuellement plusieurs fois par semaine et qui impliquent des étapes identiques à chaque fois (même commande ADUC, mêmes champs à remplir).
Analyse des processus
2
Définir les entrées et sorties du script
Lister les informations que le technicien devra saisir (inputs) et les actions que le script devra effectuer automatiquement (outputs). Par exemple : saisir le login → script réinitialise le mot de passe et déverrouille le compte.
Spécification fonctionnelle
3
Vérifier les prérequis techniques
S'assurer que le module ActiveDirectory pour PowerShell est installé sur le poste et que le technicien dispose des droits suffisants dans l'AD pour effectuer les actions ciblées.
Module : ActiveDirectory
2

Développement via PowerShell ISE

Le développement se fait dans Windows PowerShell ISE (Integrated Scripting Environment), qui offre une coloration syntaxique, une console d'exécution intégrée et un débogueur pour faciliter la mise au point.

1
Structurer le script avec des commentaires
Chaque script commence par un en-tête commenté indiquant l'auteur, la date, la version et l'objectif. Les sections importantes sont aussi commentées pour faciliter la maintenance.
Bonne pratique
2
Utiliser Read-Host pour les saisies interactives
Les informations variables (login, nom, prénom…) sont demandées dynamiquement à l'exécution via Read-Host. Le technicien n'a pas à modifier le script pour chaque utilisation.
Read-Host
3
Gérer les erreurs avec Try / Catch
Chaque appel aux cmdlets AD est encapsulé dans un bloc Try / Catch pour intercepter les erreurs (compte inexistant, droits insuffisants, mot de passe non conforme) et afficher un message explicite au technicien.
Gestion d'erreurs
💡 Les onglets ci-dessous permettent de naviguer entre les deux scripts développés.

🔑 Script — Réinitialisation mot de passe

Télécharger reset_password.ps1

👤 Script — Création utilisateur AD

Télécharger create_user.ps1
3

Test et validation du script

Avant tout déploiement, chaque script est rigoureusement testé dans un environnement de test ou sur un compte de test dédié. L'objectif est de vérifier que toutes les branches du code fonctionnent, y compris les cas d'erreur.

1
Test du cas nominal
Exécuter le script avec des données valides et vérifier dans la console ADUC que les modifications ont bien été appliquées (compte créé, mot de passe modifié, compte déverrouillé…).
Cas nominal
2
Test des cas d'erreur
Tester volontairement des cas d'erreur : login inexistant, mot de passe ne respectant pas la politique de complexité VNF, droits insuffisants. Vérifier que le message d'erreur affiché est compréhensible.
Gestion d'erreurs
3
Validation par le responsable technique
Le script est soumis au responsable technique ou au référent AD pour validation avant déploiement sur les postes des techniciens. Cette étape garantit la conformité avec les règles de sécurité VNF.
Validation avant déploiement
✅ Une fois validé, le script est sauvegardé sur le partage réseau de l'équipe Helpdesk et accessible à tous les techniciens VNF.
4

Déploiement et utilisation au quotidien

Une fois validé, le script est mis à disposition de l'équipe et intégré dans les procédures de traitement des tickets. Son utilisation devient la méthode de référence pour les tâches ciblées.

1
Mise à disposition sur le partage réseau
Le fichier .ps1 est déposé dans un dossier partagé accessible à tous les techniciens du Helpdesk VNF. Un raccourci est créé sur les bureaux des postes concernés pour un accès rapide.
Partage réseau
2
Exécution depuis PowerShell avec les droits adéquats
Le script est lancé depuis une console PowerShell ouverte en tant qu'administrateur. La politique d'exécution doit être configurée en RemoteSigned ou Unrestricted sur le poste pour autoriser l'exécution de scripts locaux.
Set-ExecutionPolicy
3
Référence dans le journal Matrix42
Lorsqu'un technicien utilise un script pour résoudre un ticket, il le mentionne dans le journal Matrix42 (ex : "Réinitialisation effectuée via script PowerShell reset_password.ps1"). Cela assure la traçabilité de l'action.
Traçabilité Matrix42
⚠️ La politique d'exécution PowerShell doit être vérifiée avant le premier lancement. Une politique trop restrictive bloquera l'exécution du script.
